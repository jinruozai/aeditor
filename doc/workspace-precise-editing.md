# Workspace Precise Editing

This document defines how agents locate and modify workspace files precisely.
It refines the existing `workspace.*` tools; it does not add a project concept,
a second permission model, or a second file API.

## Principle

The reliable editing path is:

```text
map/search -> read exact range -> edit exact text with baseHash -> validate -> verify
```

Line numbers are useful for navigation, but they are not the authority for
editing. The edit authority is the file content hash plus a unique `oldText`
anchor copied from a recent read.

## Tool Roles

```text
workspace.fileSummary   compact file tree
code.map                compact code/component outline
workspace.searchFiles   find candidate locations
workspace.readFileRange confirm the exact editable text
workspace.editFile      apply exact replacements with CAS
workspace.writeFile     create or deliberately replace whole files
workspace.patchFile     line patch escape hatch for mechanical edits
```

`workspace.editFile` is the preferred tool for changing existing source files.
`workspace.writeFile` is for new files or intentional full replacement.
`workspace.patchFile` remains useful for generated mechanical patches, but it
should not be the default for human-scale code edits because line numbers drift
when the file changes.

## Search Results

`workspace.searchFiles` returns enough information for the model to choose a
target and reread the exact range before editing:

```text
path
fileHash
line
column
matchText
previewStartLine
previewEndLine
preview
```

The search result is a locator, not an edit permission. The agent should use
`workspace.readFileRange` on the chosen area before constructing `oldText`.

## Exact Edit Contract

`workspace.editFile` accepts:

```js
{
  path: 'src/file.js',
  baseHash: 'aiditor-fnv1a-...',
  edits: [
    {
      oldText: 'exact text copied from a recent read',
      newText: 'replacement text',
      replaceAll: false
    }
  ],
  validate: 'auto'
}
```

Rules:

1. `baseHash` is required.
2. Current file hash must equal `baseHash`.
3. Each `oldText` must appear exactly once unless `replaceAll` is `true`.
4. Edits are applied atomically. If any edit fails, the file is unchanged.
5. JS and JSON validation runs before commit.
6. Successful mutations return metadata and diff summary, not full file text.

## Failure Codes

Failures must be explicit enough for the agent to recover without guessing:

```text
MISSING_BASE_HASH   editFile requires a baseHash
FILE_NOT_FOUND      target file does not exist
STALE_FILE           current hash differs from baseHash
OLD_TEXT_NOT_FOUND   oldText appears zero times
AMBIGUOUS_MATCH      oldText appears more than once
VALIDATION_FAILED    resulting file is invalid
NO_EDITS             edits list is empty
EMPTY_OLD_TEXT       an edit has no oldText anchor
NO_CHANGE            oldText and newText produce no change
```

The recommended recovery is:

```text
STALE_FILE         -> read/search again, then edit with the new hash
OLD_TEXT_NOT_FOUND -> read the nearby range and copy the exact current text
AMBIGUOUS_MATCH    -> include more surrounding context in oldText
VALIDATION_FAILED  -> inspect the produced range and repair the edit
```

## Concurrency

The compare-and-set rule is strict by default:

```text
read file@hashA
external edit changes file to hashB
editFile(baseHash: hashA)
=> reject as STALE_FILE
```

This is intentional. A stale edit should fail cleanly and be re-previewed rather
than guessing against changed content.

## Model Guidance

For existing files, agents should not start with broad rewrites. The normal
workflow is:

```text
workspace.searchFiles(...)
workspace.readFileRange(...)
workspace.editFile(...)
verify.run(...) when available
```

If the requested task needs a new file, use `workspace.writeFile`. If the task
needs many mechanical replacements generated by a patch algorithm, use
`workspace.patchFile`, still with `baseHash`.
