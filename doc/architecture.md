# AEditor Architecture

## One Mental Model

AEditor is a host for editor UI and AI-addressable actions.

```text
Core infrastructure
  -> UI framework
  -> AI host
  -> Extension packaging
```

AEditor layers depend only on lower layers:

```text
Extension
AI
UI
Core
```

Core does not know UI panels. UI does not know providers. AI does not know
domain data. Extensions do not add a second concept model.

## Architecture Invariants

These invariants are the design spine:

```text
UI identity      -> registered Component
AI action space  -> tools / context / operations
file boundary    -> workspace
reviewed changes -> operations + ChangeSet
lifecycle group  -> dotted prefix
package format   -> Extension
```

If a new feature cannot fit this list, prefer reshaping the feature before adding
a new top-level concept.

## 1. Core

Core contains stable infrastructure:

```text
signal
log
bus
history
theme
i18n
settings
commands
shortcuts
workspace
```

Core APIs must be small and generic. They should not mention game data,
animation, tables, panels generated by AI, or any domain-specific project format.

`workspace` belongs in Core because it is only a bounded read/write adapter. It
does not imply that any project has been opened. When the workspace module wants
AI access, it contributes `workspace.*` tools to the same AI tool registry used
by every other module.

## 2. UI

UI contains the editor shell and component library:

```text
component registry
dock layout data
dock runtime
component context
theme tokens
ui widgets
```

The UI layer exposes:

```js
aeditor.registerComponent(name, spec)
aeditor.resolveComponent(name)
aeditor.createDockLayout(root, config)
aeditor.ui.*
```

PanelData and toolbar items reference registered components. They are not
separate registration systems. Panel communication uses `aeditor.bus` through
the component context. Components do not directly own each other.

## 3. AI

AI contains:

```text
agent
skill
message
queue
quest
inbox
provider
transport
streaming
permission
tools
context
operations
targets
rich prompt
change set
```

AI should not contain domain-specific file formats or business rules. Domain
semantics are exposed through tools, context, and operations.

## 4. Extension

Extension is a package format and lifecycle wrapper around existing registries:

```text
components
tools
context
operations
settings
commands
styles
```

Extension install means registering those things. Extension uninstall means
removing the same name prefixes.

Commands are human/UI actions used by menus, buttons, command palettes, and
shortcuts. Tools are model-callable actions with schemas, permissions, tool-call
history, and model-visible results. A single underlying function may be exposed
as both, but the registries serve different callers.

## Naming Rules

Dotted names are the only grouping mechanism:

```text
workspace.readFile
workspace.patchFile
ui.createPanel
dock.addPanel
theme.setMode
gde.table.patchRows
ani.timeline.insertKey
```

Registry APIs should support:

```js
register(name, spec)
unregister(name)
unregisterPrefix(prefix)
get(name)
list(prefix)
```

There is no separate group field and no separate lifecycle token.

Metadata may exist for diagnostics, palette hints, review UI, and current
compatibility. It should not be the primary owner/uninstall mechanism.

## Forbidden Shape

Do not add parallel concepts that mean almost the same thing.

Bad:

```text
workspace.readFile
files.readFile
```

Good:

```text
workspace.readFile
gde.table.readRows
```

File access is generic. Domain access lives outside AEditor Core.
